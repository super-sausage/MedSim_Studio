## 一、项目总览

```
MedSim Studio
├── Frontend: React 18 + TypeScript + Vite
│   ├── 渲染引擎: Cornerstone3D (2D/MPR) + VTK.js (3D)
│   └── 状态管理: Zustand (3 个 Store)
├── Backend: Python 3.11 + FastAPI + uvicorn
│   ├── 推理引擎: TotalSegmentator + nnUNet (3 个) + MONAI
│   ├── 仿真引擎: LesionGenerator + OrganSimulator + PhantomGenerator
│   └── 存储抽象层: StorageBackend (MinIO / Local)
├── Database: PostgreSQL 16 (主) + SQLite (本地开发回退)
├── Object Storage: MinIO (S3-compatible)
├── Model Volume: Docker bind mount (4 组模型权重)
└── CI/Deploy: Docker Compose (4 服务) + NVIDIA GPU passthrough
```

**架构风格**：分层单体（Modular Monolith）—— 后端 API + 后台任务运行在同一进程内，简化开发部署。

## 二、DICOM 影像管理

### 2.1 数据模型（ORM）

三层递进：`DicomStudy` → `DicomSeries` → `DicomInstance`

| 模型 | 核心字段 | 用途 |
|---|---|---|
| `DicomStudy` | `patient_name`, `study_instance_uid`, `modalities` [JSON] | 患者级信息，搜索聚合 |
| `DicomSeries` | `series_instance_uid`, `modality`, `pixel_spacing` [JSON], `window_center/width` | 渲染参数预存 |
| `DicomInstance` | `sop_instance_uid`, `instance_number`, `image_position` [JSON], `pixel_data_path` | 切片排序 + 存储指针 |

**设计意图**：`pixel_data_path` 存的是 MinIO 对象键（`dicom/{study_uid}/{series_uid}/{sop_uid}.dcm`），而非本地路径。数据库只存元数据，影像文件全部在对象存储中，保证了后端的无状态可水平扩展。

### 2.2 上传流程（`POST /api/v1/dicom/upload`）

5 个阶段构成的完整管线：

```
Phase 0: 文件存入临时 staging 目录 (/tmp/uploads/{uuid}/)
Phase 1: 预解析 (stop_before_pixels=True) → 提取 UID → 生成 object_key
Phase 2: 全量上传到 MinIO（失败则回滚已上传的）
Phase 3: 完整 pydicom 解析 → 元数据入库
Phase 4: 清理 staging 目录
```

**关键问题一：中文 PatientName 导致解析崩溃**

- **根因**：pydicom 在缺失 `SpecificCharacterSet` tag 时无法确定编码，`str()` 抛出 `UnicodeDecodeError`
- **解法**：猴子补丁 `pydicom.dcmread`，强制所有文本 VR 类型（AE/AS/CS/DA/DS/...）在解析前预解码为 Python str，失败时回退 latin1+errors=replace

**关键问题二：上传幂等性**

- 使用 `seen_sop` set 按 SOP Instance UID 去重
- 查询数据库避免同 UID 重复入库
- Study/Series 用 `study_instance_uid` 和 `series_instance_uid` 作为自然键做 upsert

### 2.3 文件服务

`GET /instances/{id}/file` 从存储层读取 DICOM 字节流后以 `application/dicom` content-type 返回。

Cornerstone3D 通过 `wadouri:/api/v1/dicom/instances/{id}/file` URL 拉取：

```
DICOM 文件 → MinIO (字节) → Backend API (二进制 stream)
→ Cornerstone3D dicom-image-loader → Web Worker 解码 → GPU Texture
```

### 2.4 存储双后端

`StorageBackend` ABC 定义接口，`MinioStorageBackend` / `LocalStorageBackend` 通过 `factory.py` 的 `STORAGE_BACKEND` 环境变量切换。

核心接口：

- `upload_file(object_key, file_path, content_type)`
- `get_object_bytes(object_key)`
- `delete_file(object_key)`
- `check_health()`

## 三、Cornerstone3D 渲染引擎

渲染引擎分 5 个阶段逐步构建：

### 3.1 Phase 1: 初始化（`initCornerstone.ts`）

单例模式，仅初始化一次：

```typescript
cs.setUseSharedArrayBuffer(false);       // 避免跨域隔离头
await cs.init(RENDERING_CONFIG);          // WebGL 初始化
csToolsInit();                             // 工具事件系统
dicomImageLoaderExternal.cornerstone = cs;
dicomImageLoaderExternal.dicomParser = await import('dicom-parser');
configureImageLoader({maxWebWorkers: Math.min(hardwareConcurrency, 4)});
addTool(WindowLevelTool);                 // 注册 10 种工具
```

**为何 `setUseSharedArrayBuffer(false)`？** SharedArrayBuffer 需要 `Cross-Origin-Opener-Policy` 和 `Cross-Origin-Embedder-Policy` 头，Docker nginx 通常没有配置，所以回退到常规 typed array。

### 3.2 Phase 2: 渲染引擎（`createRenderingEngine.ts`）

单例 `RenderingEngine`（ID: `ctViewerEngine`）管理所有 Viewport：

- `enableStackViewport()` — 2D 切片模式，一次显示一张 slice
- `enableVolumeViewport()` — 3D 体模式，支持 ORTHOGRAPHIC 和 orientation 指定

核心：三个 MPR 方向（轴向/矢状/冠状）共享同一个 `createAndCacheVolumeFromImages` 构建的 3D Volume，浏览切片只是改变渲染平面的位置。

### 3.3 Phase 3: 数据加载（`loadDicomSeries.ts`）

```
fetchImageIdsForSeries(seriesId)
  → GET /api/v1/dicom/series/{seriesId}/instances
  → 构建 imageId[]: "wadouri:/api/v1/dicom/instances/{id}/file"

loadVolumeOnViewport(viewport, seriesId, volumeId, opts)
  → createAndCacheVolumeFromImages(volumeId, imageIds)
```

### 3.4 Phase 4: Viewport 组件（`CornerstoneViewport.tsx`）

React 封装的关键设计：

- **3 种 UI 状态**：`loading` / `loaded` / `error`，各有对应 UI
- **ResizeObserver** 监听容器尺寸变化 → `resizeViewports()`
- **StrictMode 安全**：`cancelled` flag 防止双初始化竞态
- **清理**：`disableViewport(viewportId)` 释放 WebGL 资源和事件监听

Volume vs Stack 的分支逻辑：

```typescript
const isVolume = !!orientation;
if (isVolume) {
  viewport = await enableVolumeViewport({ viewportId, element, orientation });
  await loadVolumeOnViewport(viewport, seriesId, volumeId);
} else {
  viewport = await enableStackViewport({ viewportId, element });
  await loadSeriesOnViewport(viewportId, seriesId);
}
```

### 3.5 Phase 5: 工具系统（`toolGroups.ts`）

```
Left Mouse   → WindowLevelTool
Middle Mouse → PanTool
Right Mouse  → ZoomTool
Scroll Wheel → StackScrollMouseWheelTool
```

ViewerPage 将 3 个 viewport 渲染在 `grid-cols-2 grid-rows-2` 布局中（冠状位占 col-span-2），工具栏控制工具切换 + 窗宽窗位预设。

## 四、病灶仿真生成

### 4.1 LesionGenerator（`lesion/generator.py`）

**核心算法——距离场 + Sigmoid 软边缘**：

```python
distance = sqrt((z-cz)/rz)^2 + ((y-cy)/ry)^2 + ((x-cx)/rx)^2)

# 形状变形
if shape_type == "lobulated":
    distance = apply_lobulation(distance, ...)   # 多正弦波叠加
if shape_type == "spiculated":
    distance = apply_spiculation(distance, ...)  # 表面加权变形
if shape_type == "irregular":
    distance = apply_irregularity(distance, ...) # Perlin 噪声

# Sigmoid 软边缘
mask = 1 / (1 + exp((distance - 1.0) / (margin_width * 0.2)))
hu_volume = gaussian_noise(hu_mean, hu_std) * mask
```

**5 种病变类型与默认 HU 值**：

| 类型 | 均值 HU | 标准差 | 临床对应 |
|---|---|---|---|
| tumor | 40 | 20 | 软组织肿瘤 |
| nodule | -100 | 50 | 肺结节（半实性） |
| cyst | 10 | 5 | 囊肿（近水密度） |
| calcification | 300 | 100 | 钙化（高密度） |
| metastasis | 50 | 25 | 转移灶（中等密度+高变异） |

**4 种形状的数学实现**：

- **Spherical**：标准椭球，无变形
- **Lobulated（分叶状）**：3 层正弦波乘积叠加，`distance = distance / (1.0 + deformation)`，幅度 0.2~0.5
- **Spiculated（毛刺状）**——最有技术含量的实现：
  - 球坐标 `θ = arctan2(y-cy, x-cx)`, `φ = arctan2(z-cz, sqrt(...))`
  - 8~16 根毛刺，每根为 8 次幂窄角 lobe：`max(cos(θ-θi)cosφ, 0)^8`
  - **表面加权**：`surface_weight = exp(-((distance-1.0)/length)^2)`，变形仅作用于边界附近
  - 用**减法**（非除法）将表面向外推
- **Irregular（不规则）**：Perlin 噪声（均匀分布 → sigma=1.5 高斯模糊），`distance *= (1.0 + noise)`

### 4.2 OrganSimulator（`organ/simulator.py`）

参数化椭球体 + HU 分布 + 可选增强模式：

```python
distance = sqrt((z/rz)^2 + (y/ry)^2 + (x/rx)^2)
mask = gaussian_filter((distance <= 1.0).astype(float), sigma=2) > 0.3
```

器官尺寸相对于 volume 比例定义（如 liver: `radii=(dz*0.3, dy*0.25, dx*0.35)`）。

**增强模式**：

| 模式 | 实现方式 |
|---|---|
| homogeneous | 整体增加 HU |
| heterogeneous | 高斯模糊噪声贴图制造不均匀增强 |
| rim | 距离变换提取边缘 5 像素环 |
| septal | 内置分隔条增强（骨架实现） |

### 4.3 PhantomGenerator（`phantom_generator.py`）

**Procedural（程序化生成）**：几何基元（椭圆、弧线）拼合近似人体上腹部的 CT 断面。适合 UI demo，**非医学图像**。

**Atlas（图谱驱动）**：读取 `models/phantom_atlas/{case_id}/` 下的真实 CT NIfTI：
- 自动检测 Z 轴朝向（从 NIfTI affine 矩阵读取），不匹配则翻转
- 按最大边长等比例重采样
- 返回 Float32 CT + Uint8 标签图（base64 编码）

### 4.4 后台任务（`run_simulation_job`）

状态机：`pending → running → completed / failed`

```
Step 1: Build base volume (progress 10% → 25%)
  └─ 从 DICOM 构建 或 合成空 volume

Step 2: Apply overlays (progress 25% → 50%)
  ├─ 遍历 LesionConfig → generate_lesion() → 写入
  └─ 遍历 OrganConfig → generate_organ() → 仅覆盖背景

Step 3: Write NRRD (progress 50% → 75%)
  └─ SimpleITK, 注意 (z,y,x) ↔ (x,y,z) 坐标转换

Step 4: Upload to MinIO (progress 75% → 90%)

Step 5: Mark completed (progress 90% → 100%)
```

注意：`run_simulation_job()` 使用**独立的数据库 session**（`SessionLocal()`），不沿用请求 session——避免跨线程 session 重用。

### 4.5 实时预览

- `POST /preview/lesion` — 32³ 快速预览，体素统计
- `POST /preview/lesion-on-dicom` — 在真实 DICOM 上生成病灶，返回 side-by-side PNG（base64）+ HU 统计
- `POST /debug-lesion` — 全诊断管线：生成统计 + 写入验证 + 位置校验 + 间距分析 + 预览 PNG

## 五、AI 分割流水线

### 5.1 编排层（`segmentation/ai/pipeline.py`）

```
run_full_segmentation(job_id, storage, instances, model_name, ...)
  │
  ├─ build_volume_from_dicom()
  │
  ├─ 根据 model_name 路由:
  │   ├─ nnunet_lung_lobe → Dataset703 (5 肺叶)
  │   ├─ nnunet_handoff → Dataset701 (6 器官)
  │   ├─ nnunet702_20organs → Dataset702 (20 类)
  │   ├─ totalsegmentator → TotalSegmentator v2 (117 结构)
  │   └─ unet/segresnet/swin_unetr → MONAI 推理
  │
  ├─ 可选: run_lesion_detection() [MONAI 路径]
  │
  └─ export_mask_nrrd() → upload to MinIO
```

所有 nnUNet 模型遵循统一推理模板：

1. 导出体积为临时 NIfTI
2. `nnUNetPredictor.predict_single_npy_array()`（输入 `(1, W, H, D)`）
3. 结果转置回 `(Z, Y, X)`，返回 int32 标签图

### 5.2 模型矩阵

| 模型名 | 数据源 | 类别数 | 典型路径 |
|---|---|---|---|
| `totalsegmentator` | pip 包预训练 | 117 | `TOTALSEGMENTATOR_DIR` |
| `nnunet_handoff` | Dataset701（自训练） | 6 | `NNUNET_CUSTOM_MODEL_PATH` |
| `nnunet702_20organs` | Dataset702（自训练） | 20 | `NNUNET_CUSTOM_20_MODEL_PATH` |
| `nnunet_lung_lobe` | Dataset703（自训练） | 5 | `NNUNET_LUNG_LOBE_MODEL_PATH` |
| `unet` | `unet.pth`（19MB） | 5+ | 代码内路径 |
| `segresnet/swin_unetr` | MONAI zoo | 可变 | 按需加载 |

### 5.3 交互式精修（`refine_mask_on_click`）

基于**强度约束的区域生长**：

1. 用户在 slice 上点击 `(z, x, y)`
2. 从该点出发以局部 HU 范围为约束做连通生长
3. 将结果 add/remove 到当前 label_map
4. 更新后的 mask 写回 MinIO

### 5.4 标签系统（`ai/totalsegmentator/labels.py`）

统一标签管理，支持多模型标签合并：

```python
get_label_defs(model_name, categories=False)
```

117 类结构定义：

- `index` — 整数标签索引
- `name` — 结构名称（英文）
- `color` — RGBA 颜色元组（前端覆盖渲染用）
- `category` — 分类（organs/bones/vessels/muscles/glands）
- `category_label` — 中文分类名

## 六、存储与部署架构

### 6.1 存储分层

```
请求流量
    │
    ▼
┌─────────────────────────────┐
│ PostgreSQL (元数据)          │
│ • DicomStudy / DicomSeries  │  ← 模式: Alembic 迁移
│ • DicomInstance             │
│ • SegmentationJob           │
│ • SimulationJob             │
└─────────────────────────────┘
    │
    ▼
┌─────────────────────────────┐
│ MinIO (对象存储)             │
│ • dicom/{study}/{ser}/...   │
│ • segmentation/{job}/...    │
│ • simulation/{job}/...      │
│                             │
│ LocalStorage (回退)          │  ← STORAGE_BACKEND=local
└─────────────────────────────┘
```

**设计意图**：元数据与大文件分离，数据库保持轻量，后段实例可无状态水平扩展。

### 6.2 对象键约定

```
dicom/{study_uid}/{series_uid}/{sop_uid}.dcm
segmentation/{job_id}/mask.nrrd
segmentation/{job_id}/labels.json
simulation/{job_id}/result.nrrd
```

### 6.3 Docker Compose 架构

4 个服务通过 `ct-network` 互通：

```
ct-frontend (nginx:80 → 映射 5173)
  │
  └─→ ct-backend (uvicorn:8000, 2 workers, GPU passthrough, shm=8g)
        │
        ├─→ ct-postgres (postgres:16-alpine, 5432)
        │
        └─→ ct-minio (minio:9000 API, 9001 Console)
```

**Volume 设计**：

| Volume | 类型 | 用途 |
|---|---|---|
| `postgres_data` | 命名 volume | 数据库持久化 |
| `minio_data` | 命名 volume | MinIO 对象持久化 |
| `backend_uploads` | 命名 volume | 临时上传目录 |
| `backend_models` | 命名 volume | AI 模型缓存 |
| `backend_dicom` | 命名 volume | 本地 DICOM 文件 |

**模型权重**均使用 **bind mount**（`./models/totalsegmentator` 等），保证大数据不写入 Docker layer，本地更新即时生效。

**GPU 配置**：

```yaml
deploy:
  resources:
    reservations:
      devices:
        - driver: nvidia
          count: 1
          capabilities: [gpu]
```

**shm_size = 8g**：TotalSegmentator 的 PyTorch DataLoader 多进程 workers 需要大量 `/dev/shm` 做 IPC。默认 64MB 远不够，需要 8GB。

### 6.4 配置体系

`backend/app/core/config.py` 使用 `pydantic-settings` 从环境变量读取，支持 `.env` 文件覆盖：

```python
# Docker 典型值
DATABASE_URL=postgresql://ctuser:ctpass123@postgres:5432/ct_simulator
STORAGE_BACKEND=minio
AI_DEVICE=cuda

# 本地开发回退
DATABASE_URL=sqlite:///./ct_simulator.db
STORAGE_BACKEND=local
AI_DEVICE=cpu
```

## 七、前端状态管理架构

### 7.1 Zustand 三 Store 设计

**选择 Zustand 而非 Redux**：零样板代码、TypeScript 原生支持、轻量级（仅管理客户端 UI 状态）。

| Store | 核心状态 | 设计意图 |
|---|---|---|
| `useViewerStore` | 活动 tool、MPR 视图参数、窗宽窗位预设、体渲染开关 | 纯 UI 状态，不需持久化 |
| `useSimulationStore` | 病灶/器官配置列表、任务列表、选中状态 | 仿真 UI 的 CRUD 操作 |
| `useSegmentationStore` | 模型列表、标签集、器官选择、任务轮询 | 分割任务生命周期管理 |

**Store 分离理由**：

- Segmentation 需要轮询（2s interval）+ 蒙版数据持久化 + 模型/标签发现
- Simulation 是配置驱动（病灶列表 + 器官列表），任务提交一次性
- 合并会导致单 store 膨胀，更新频繁且互相干扰

### 7.2 API 服务层（`api.ts`）

```typescript
// 请求拦截器: camelCase → snake_case
// 响应拦截器: snake_case → camelCase
// 错误归一化: { status, message, data }
```

这一层是前后端对接的"翻译器"。后端 snake_case（Python 约定），前端 camelCase（TypeScript 约定），拦截器自动双向转换。前端代码里所有调用都用 `studyId`、`seriesId`，HTTP 请求体里自动变成 `study_id`、`series_id`。

### 7.3 生命周期管理（以 Segmentation 为例）

```
用户点击"开始分割"
  → createJob(config)
    → segmentationService.createJob(payload)
      → POST /api/v1/segment/jobs → { id, status: "pending" }
    → store.addJob(response)
  → startPolling(jobId)
    → setInterval (2s) → GET /api/v1/segment/jobs/{id}
      → updateJobStatus()
      → status === "completed" → stopPolling()
```

### 7.4 蒙版数据持久化

```typescript
interface PersistedMaskData {
  jobId: string;
  seriesId: string;
  maskData: Float32Array | null;
  labels: SegmentationLabel[];
}
```

**设计意图**：用户在 SegmentationPage 完成分割后切换到 ViewerPage 查看，再返回 SegmentationPage 时蒙版数据仍在 store 中——不需要重新下载。
